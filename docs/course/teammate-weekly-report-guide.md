# 队友周报写作参考

说明：

- 前三部分建议与个人周报保持一致，只调整每周小组分工中的个人侧重点。
- 第四、第五部分要突出队友在检索、评测、实验和案例整理上的贡献。
- 以下文本为队友可参考版本。

## 第 1 周队友版重点

四、本人负责的部分

本人本周主要负责项目环境配置、依赖安装、数据集加载和基础评测流程测试。具体包括根据 starter kit 文档配置运行环境，检查 `requirements.txt`、`local_evaluation.py` 和检索相关依赖，尝试加载 CRAG-MM 数据集，并测试图像检索 API 的基本调用方式。同时，我负责记录环境配置中遇到的问题、运行命令和初始 baseline 的输出情况，为后续任务1实验提供基础。

五、针对本人负责的部分，所做的工作及完成情况，反思等

本周我完成了项目运行环境的初步配置，并对数据加载和图像检索流程进行了测试。通过实际运行，我了解了 CRAG-MM 数据集中图片、问题、答案和 interaction_id 的组织方式，也熟悉了本地评测脚本的主要参数。当前主要问题是依赖较多，部分模型和检索组件对本地计算资源要求较高，后续需要继续记录稳定的运行方式，并配合组内完成任务1 baseline 的小规模评测和错误案例整理。

## 第 2 周队友版重点

四、本人负责的部分

本人本周主要负责网页检索 API 接入、任务1和任务2的小规模评测，以及中期汇报中实验结果部分的准备。我重点测试了 web search 返回结果的字段，包括网页标题、摘要、链接和相关性分数，并整理了图像检索与网页检索结果在任务2中的使用方式。同时，我负责记录不同样例下模型的回答表现，区分正确、错误和信息不足三类情况。

五、针对本人负责的部分，所做的工作及完成情况，反思等

本周我完成了网页检索 API 的初步接入，并配合完成任务2多源增强原型。通过对比任务1和任务2的样例，我发现网页检索能够补充图像 KG 中缺失的信息，但也会带来噪声和无关内容。因此，仅增加检索源并不能自动提升效果，还需要对检索结果进行筛选和组织。在中期汇报准备中，我整理了部分典型案例，用于说明系统当前的优势和问题。后续将继续完善实验记录和错误分析表。

## 第 3 周队友版重点

四、本人负责的部分

本人本周主要负责任务2不同版本的对比实验、错误案例整理和任务3评测流程接入。我围绕多源检索结果的数量、查询构造方式和回答效果进行记录，整理了若干成功与失败案例。同时，我开始测试多轮数据集，了解 `message_histories` 在评测脚本中的构造方式，并记录多轮问答中容易连续出错的样例。

五、针对本人负责的部分，所做的工作及完成情况，反思等

本周我完成了任务2部分版本的实验对比，并初步接入任务3评测流程。实验表明，模型错误不一定来自生成阶段，也可能来自检索阶段没有命中关键信息，或者检索结果虽然包含答案但被无关内容干扰。因此我在错误分析中增加了“检索是否命中”和“模型是否正确利用证据”两个维度。任务3测试中，我发现多轮问题对历史顺序和前序回答质量要求较高，后续需要继续整理连续错误案例。

## 第 4 周队友版重点

四、本人负责的部分

本人本周主要负责最终实验结果汇总、评测指标整理、案例分析和源码提交检查。我将前三个任务的运行结果进行归纳，整理 baseline、任务1版本、任务2版本和任务3版本的表现差异，并为最终报告和 PPT 准备实验表格与典型案例。同时，我检查了源码结构、依赖文件、运行命令和提交材料，确保最终验收材料较为完整。

五、针对本人负责的部分，所做的工作及完成情况，反思等

本周我完成了实验结果和案例材料的最终整理，并配合完成总结汇报准备。通过汇总三个任务的结果，我认识到多模态 RAG 系统的性能不仅取决于模型能力，还与检索质量、证据筛选和评测规则密切相关。尤其在多轮任务中，前序错误会影响后续回答，因此实验分析不能只看单个问题，还要关注整段对话的稳定性。本次实训提升了我对检索增强生成系统评测和错误定位的理解。
